# Design Assistant v3 Redesign

## Goal

Agent 백엔드를 처음부터 재설계하여 중첩 agent 구조로 인한 반복 버그(interrupt 전파 실패, streaming 필터링, 답변 중복, 컨텍스트 오염)를 구조적으로 제거한다.

## Architecture

LangGraph.js 공식 패턴 기반 Supervisor + ReAct Agent 노드 구조.

**핵심 변경:**
1. `streamEvents` → `stream()` with `["messages", "custom", "updates"]`
2. 중첩 ReAct 제거 → Supervisor(라우팅) + flat agent 노드
3. Research Agent를 Planner + Worker로 분리 (컨텍스트 오염 방지)
4. 매 메시지 새 thread_id → 세션당 고정 thread_id (MemorySaver)
5. ConversationManager 삭제

## Tech Stack

- LangGraph.js (`@langchain/langgraph`)
- `createReactAgent` (`@langchain/langgraph/prebuilt`)
- `Command` for agent routing
- `MemorySaver` for conversation persistence
- `interrupt()` for HITL
- `config.writer()` for custom streaming
- Langfuse (optional observability)
- Electron IPC (streaming bridge)
- Vue 3 + Pinia (frontend, 최소 수정)

## State Schema

```typescript
import { Annotation, MessagesAnnotation } from "@langchain/langgraph";

const GraphAnnotation = Annotation.Root({
  ...MessagesAnnotation.spec,
  searchEnabled: Annotation<boolean>({ default: () => true }),
  conversationSummary: Annotation<string>({ default: () => "" }),
});
```

- `messages`: 대화 히스토리 (MessagesAnnotation 상속)
- `searchEnabled`: 검색 토글 상태. Supervisor가 라우팅 시 참조. `configurable`이 아닌 state로 관리하여 graph input으로 전달 가능.
- `conversationSummary`: 오래된 대화의 LLM 요약. 임계치 초과 시 자동 생성.

`searchEnabled`는 Electron IPC를 통해 graph input에 포함:
```typescript
await graph.stream(
  { messages: [new HumanMessage(text)], searchEnabled },
  { configurable: { thread_id: threadId } }
);
```

## Graph Structure

```
User Message
    ↓
┌──────────────────────┐
│ Supervisor           │
│ (structuredOutput)   │
│ → next: agent name   │
└──────────┬───────────┘
           ↓ Command({ goto: next })
┌──────────┴──────────────────────────────┐
│          │              │               │
│ research │  pc_fix      │  chat         │
│ (Planner │  (ReAct)     │  (ReAct)      │
│  ReAct)  │  diag tools  │  no tools     │
│          │  script_runner│               │
└──────────┴──────────────┴───────────────┘
           ↓ Command({ goto: "supervisor" })
┌──────────────────────┐
│ Supervisor           │
│ → "__end__" or next  │
└──────────────────────┘
```

### Supervisor Node

- 역할: 라우팅만 담당. 답변 생성 안 함.
- `llm.withStructuredOutput()`으로 `{ next: "research" | "pc_fix" | "chat" | "__end__" }` 반환
- `searchEnabled === false`이면 research 선택 불가 → chat으로 fallback
- `Command({ goto: response.next })`로 agent 전환
- structured output이므로 JSON 토큰이 streaming에 노출되지 않음

### Agent 노드 통합 방식

각 agent는 `createReactAgent`로 생성하되, **wrapper 함수를 통해 그래프 노드로 등록**한다. wrapper가 agent 실행 후 `Command({ goto: "supervisor" })`를 반환:

```typescript
const researchAgent = createReactAgent({ llm, tools, prompt });

async function researchNode(state: typeof GraphAnnotation.State) {
  const result = await researchAgent.invoke(state);
  const lastMsg = result.messages[result.messages.length - 1];
  return new Command({
    goto: "supervisor",
    update: { messages: [lastMsg] },
  });
}

graph.addNode("research", researchNode, { ends: ["supervisor", "__end__"] });
```

이렇게 하면:
- `createReactAgent`의 interrupt가 부모 그래프로 전파 (inherited checkpointer)
- wrapper에서 `Command`로 명시적 라우팅
- agent 내부 구조는 캡슐화

### Research Agent (Planner)

- 역할: 질문 분해 → 순차 조사 → 종합 답변 생성
- `createReactAgent` with tools: `[research_worker, ask_user]`
- 복합 질문을 단계별로 분해하여 `research_worker`를 반복 호출
- 각 worker 호출은 깨끗한 컨텍스트에서 실행 → 이전 단계 raw 데이터가 Planner에 누적 안 됨
- `ask_user`로 모호한 질문 시 사용자에게 보충 질문 (interrupt)
- `config.writer()`로 진행상황(현재 단계, 키워드 등) custom stream
- `recursionLimit: 25` (기본 Planner 루프 제한)

### Research Worker (Tool)

- 역할: 단일 조사 작업 수행. Planner가 호출하는 tool.
- 내부적으로 wiki 순수 함수 사용: `searchWikipedia()`, `getSummary()`, `getDetail()`
- 검색 → 관련 문서 판단 → 요약/상세 조회 → 결과 string 반환
- 매 호출마다 독립적 실행, 컨텍스트 공유 없음
- tool 함수의 두 번째 인자 `config: LangGraphRunnableConfig`에서 `config.writer` 접근:

```typescript
import { LangGraphRunnableConfig } from "@langchain/langgraph";

const researchWorkerTool = tool(
  async ({ query }: { query: string }, config: LangGraphRunnableConfig) => {
    config.writer?.({ type: "search_start", query });
    const results = await searchWikipedia(query);
    config.writer?.({ type: "search_result", titles: results.map(r => r.title) });
    // ... 요약/상세 조회
    return JSON.stringify({ summary, sources });
  },
  { name: "research_worker", ... }
);
```

### PC Fix Agent

- 역할: PC 진단 + 스크립트 실행
- `createReactAgent` with tools: `[system_info, network_check, installed_programs, script_runner]`
- `script_runner` 내부에서 `interrupt()`로 실행 전 사용자 확인
- wrapper 함수에서 `Command({ goto: "supervisor" })`로 완료 후 반환
- `recursionLimit: 15`

### Chat Agent

- 역할: 일반 대화 (인사, 후속 질문, 짧은 반응)
- tool 없이 단순 LLM 호출 노드 함수로 구현 (`createReactAgent` 오버헤드 제거):

```typescript
async function chatNode(state: typeof GraphAnnotation.State) {
  const response = await llm.invoke([
    new SystemMessage(CHAT_PROMPT),
    ...state.messages.slice(-WINDOW_SIZE),
  ]);
  return new Command({
    goto: "supervisor",
    update: { messages: [response] },
  });
}
```

## Streaming

### Stream Modes

```typescript
graph.stream(input, {
  configurable: { thread_id: threadId },
  streamMode: ["messages", "custom", "updates"],
  callbacks,  // Langfuse tracer (optional)
});
```

| Mode | 용도 | 데이터 |
|---|---|---|
| `messages` | LLM 토큰 실시간 전달 | `[AIMessageChunk, { langgraph_node }]` |
| `custom` | tool 내부 진행상황 | `config.writer()`로 전송한 임의 데이터 |
| `updates` | 노드 완료/전환 감지 | state 변경사항, interrupt 감지 |

### Custom Stream 데이터 타입

```typescript
// research_worker 내부에서
config.writer?.({ type: "search_start", query: "Computer-aided design" });
config.writer?.({ type: "search_result", titles: ["CAD", "Ivan Sutherland"], count: 2 });
config.writer?.({ type: "source_found", title: "CAD", url: "https://...", snippet: "..." });
config.writer?.({ type: "research_step", step: "CAD 핵심 인물 조사 중..." });
```

### IPC Events (Electron)

```
electron/main.ts → renderer:
  'stream:token'     ← messages mode (agent LLM 토큰)
  'stream:custom'    ← custom mode (키워드, 출처, 진행상황)
  'stream:done'      ← stream 종료 + 최종 결과
  'stream:error'     ← 에러
  'stream:interrupt'  ← interrupt 감지 (ask_user / script confirm)
```

### Interrupt 처리

```typescript
// interrupt 감지: updates mode에서 "__interrupt__" 확인
// 또는 stream 종료 후 getState()로 확인

// resume:
await graph.stream(new Command({ resume: userResponse }), config);
```

- interrupt는 inherited checkpointer로 서브그래프에서 부모로 자동 전파
- 각 agent(research, pc_fix)에서 직접 `interrupt()` 호출
- electron/main.ts에서 interrupt 감지 → renderer에 전달 → 사용자 응답 → resume

## Conversation History

### MemorySaver + 고정 thread_id

```typescript
const checkpointer = new MemorySaver();
const graph = stateGraph.compile({ checkpointer });

// 앱 시작 또는 리셋 시 새 thread_id 생성
let threadId = `session-${Date.now()}`;

// 모든 메시지가 같은 thread → 이전 대화 자동 포함
await graph.stream(
  { messages: [new HumanMessage(userInput)] },
  { configurable: { thread_id: threadId } }
);
```

### 컨텍스트 윈도우 관리: 요약 + 최근 N개

대화가 길어지면 오래된 메시지를 LLM으로 요약하고, 최근 N개는 원문 유지.

#### Supervisor에서 요약 트리거

Supervisor가 실행될 때마다 메시지 수를 체크. 임계치(WINDOW_SIZE * 2) 초과 시 오래된 메시지를 요약 후 제거:

```typescript
const WINDOW_SIZE = parseInt(process.env.CONVERSATION_WINDOW_SIZE || '10');

async function supervisor(state: typeof GraphAnnotation.State) {
  if (state.messages.length > WINDOW_SIZE * 2) {
    const oldMessages = state.messages.slice(0, -WINDOW_SIZE);
    const summaryResponse = await llm.invoke([
      new SystemMessage("이전 대화 내용을 간결하게 요약하세요. 핵심 정보와 맥락만 포함."),
      ...(state.conversationSummary
        ? [new SystemMessage(`기존 요약: ${state.conversationSummary}`)]
        : []),
      ...oldMessages,
    ]);

    // 라우팅 + 요약 업데이트 + 오래된 메시지 제거
    const route = await routeLLM.invoke(state.messages);
    return new Command({
      goto: route.next,
      update: {
        conversationSummary: String(summaryResponse.content),
        messages: state.messages.slice(-WINDOW_SIZE),
      },
    });
  }

  // 임계치 미만: 정상 라우팅
  const route = await routeLLM.invoke(state.messages);
  return new Command({ goto: route.next });
}
```

#### Agent prompt에서 요약 포함

각 agent의 `prompt` 함수에서 요약을 시스템 메시지로 포함:

```typescript
const agent = createReactAgent({
  llm,
  tools,
  prompt: (state) => [
    new SystemMessage(SYSTEM_PROMPT),
    ...(state.conversationSummary
      ? [new SystemMessage(`[이전 대화 요약]\n${state.conversationSummary}`)]
      : []),
    ...state.messages.slice(-WINDOW_SIZE),
  ],
});
```

- 요약은 Supervisor 노드에서 **임계치 초과 시 한 번만** 실행 (매 LLM 호출마다 아님)
- 최근 N개 메시지는 원문 유지 → 디테일 보존
- 오래된 대화는 요약으로 압축 → 맥락 유지 + 토큰 절약
- 기존 요약이 있으면 누적 요약 (새 요약 = 기존 요약 + 새로 밀려난 메시지)
- `messageModifier`는 deprecated → `prompt` 함수 사용

### 리셋

```typescript
// 새 thread_id 생성으로 리셋 (UUID 사용으로 충돌 방지)
import { randomUUID } from 'crypto';
threadId = `session-${randomUUID()}`;
```

ConversationManager, LLM 요약, 수동 히스토리 관리 모두 삭제.

### Langfuse (Optional)

```typescript
// agent/observability.ts
import { CallbackHandler } from 'langfuse-langchain';

export function createTracer(): CallbackHandler | null {
  if (!process.env.LANGFUSE_SECRET_KEY) return null;
  return new CallbackHandler({
    secretKey: process.env.LANGFUSE_SECRET_KEY,
```

- 다른 tracer(LangSmith 등)로 교체 시 이 파일만 수정

## Error Handling

### LLM/Tool 에러

```typescript
// electron/main.ts - stream 소비 시
try {
  for await (const [mode, chunk] of stream) {
    // ... 정상 처리
  }
} catch (error) {
  // GraphInterrupt는 에러가 아님 → interrupt 처리로 분기
  if (error.name === 'GraphInterrupt') {
    // stream:interrupt 전송
  } else {
    // stream:error 전송 → UI에서 ErrorRetryCard 표시
    webContents.send('stream:error', { message: error.message });
  }
}
```

### 에러 전략

| 상황 | 처리 |
|---|---|
| LLM API 호출 실패 | stream:error → UI에 표시, 사용자가 retry |
| Tool 타임아웃 (wiki API) | tool 내부에서 try/catch → 에러 string 반환 → agent가 다른 접근 시도 |
| Graph interrupt | interrupt 데이터 파싱 → stream:interrupt 전송 |
| 알 수 없는 에러 | stream:error → UI에 표시, thread 상태 유지 (resume 가능) |

### Retry

- 자동 retry 없음. 사용자가 ErrorRetryCard에서 retry 클릭 시 같은 메시지 재전송.
- thread_id가 유지되므로 이전 대화 컨텍스트는 보존.

## Preload Bridge Changes

IPC 채널 변경에 따른 preload.js 업데이트:

```
// 변경되는 채널
stream:step      → stream:custom     (custom stream mode 데이터)
agent:message    → agent:message     (유지)
agent:reset      → agent:reset       (유지)

// 새 채널
stream:interrupt  (ask_user / script confirm)

// 삭제
stream:step      (stream:custom으로 대체)
```

preload.js의 `contextBridge.exposeInMainWorld` 업데이트:
- `onStreamStep` → `onStreamCustom`
- `onStreamInterrupt` 추가
- 나머지(`onStreamToken`, `onStreamDone`, `onStreamError`, `sendConfirmResponse`, `sendClarifyResponse`) 유지

## Key Design Decisions

| 결정 | 이유 |
|---|---|
| `streamEvents` 대신 `stream()` 사용 | 공식 패턴. mode별 분리로 필터링 로직 제거 |
| Supervisor를 structuredOutput으로 | JSON 파싱 안정성, streaming에 토큰 노출 안 됨 |
| Research를 Planner + Worker 분리 | 순차 조사 시 컨텍스트 오염 구조적 방지 |
| wiki 함수를 tool → 순수 함수로 | Worker 내부에서만 사용, tool 오버헤드 제거 |
| generate_answer tool 삭제 | Research Agent가 직접 답변, 3중 구조 제거 |
| ConversationManager 삭제 | MemorySaver + thread_id로 충분 |
| Langfuse를 callbacks로 주입 | 단일 파일 격리, env 없으면 비활성, 교체 용이 |
| 프론트엔드 최소 수정 | 백엔드 안정화 우선, IPC 인터페이스만 맞춤 |

## Data Source Extensibility

현재 Wikipedia만 사용. 나중에 다른 소스 추가 시:
1. `agent/tools/`에 새 API 함수 파일 추가 (예: `google-search-api.ts`)
2. `research-worker.ts`에서 해당 함수를 조합하여 사용
3. 또는 별도 worker tool 추가 (예: `google_research_worker`)
4. Research Agent(Planner)의 tool 목록에 추가

agent 구조나 graph 변경 없이 tool 추가만으로 확장 가능.
